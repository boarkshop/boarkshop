# MVP

> Этот документ дополняет `DESIGN.md` и не является самостоятельным описанием системы. Термины, общая архитектура и назначение Boarkshop определяются в `DESIGN.md`; здесь фиксируются только границы первой реализации.

## Цель

MVP должен доказать один полный сценарий:

```text
событие из HTTP / Telegram / Cron
→ транспортная нормализация
→ проверка guard каждого активного пайплайна
→ последовательное выполнение принятых пайплайнов
→ обмен данными через файловые директории запуска
```

MVP не должен доказывать готовность платформы к промышленной эксплуатации. Его задача — зафиксировать устойчивые контракты ядра, вокруг которых затем можно наращивать управление пайплайнами, надёжность, безопасность и дистрибуцию.

## Граница MVP

```text
Boarkshop daemon
├── listeners
│   ├── generic HTTP
│   ├── Telegram long polling
│   └── Cron
├── bounded in-memory dispatcher
└── pipeline executor
    └── guard → step 1 → step 2 → ... → step N
```

Dispatcher является внутренней частью runtime, а не отдельной пользовательской подсистемой. Он отделяет приём событий от выполнения команд и ограничивает параллелизм.

## Что входит

### 1. Демон и статическая конфигурация

MVP работает как один процесс на одном сервере.

Он должен поддерживать:

- запуск из одного автономного бинарного файла;
- загрузку конфигурации при старте;
- загрузку пайплайнов из локальной директории;
- валидацию конфигурации и активных пайплайнов до начала приёма событий;
- корректное завершение с остановкой listeners и запущенных процессов;
- структурированные логи с `event_id`, `run_id`, `pipeline_id`, `step_id`, длительностью и exit code.

Изменение конфигурации применяется через перезапуск демона. Hot reload в MVP не требуется.

### 2. Общий контракт событий

Каждое событие должно содержать как минимум:

- `source`;
- `schema_version`;
- `event_id`;
- `received_at`.

Остальная схема определяется конкретным listener-ом. Часто используемые для фильтрации транспортные поля размещаются в корне события, а исходное представление сохраняется во вложенном объекте.

Схемы всех трёх listeners должны быть зафиксированы до реализации. В пределах одной версии допускаются только обратно совместимые добавления полей. Несовместимое изменение требует новой `schema_version`.

Listener не знает о пайплайнах и не содержит бизнес-логики интеграций. Его единственный выход — передача нормализованного события в dispatcher.

### 3. HTTP listener

HTTP listener должен:

- слушать один настраиваемый адрес;
- принимать любые HTTP-методы и пути;
- извлекать method, path, query, headers, remote address и body;
- иметь настраиваемый предел размера body;
- передавать событие в dispatcher без ожидания выполнения пайплайнов.

После принятия события listener возвращает фиксированный транспортный ответ `202 Accepted`. Результаты пайплайнов не влияют на HTTP-ответ. При переполнении внутренней очереди допускается `503 Service Unavailable`, при превышении лимита body — `413 Content Too Large`.

### 4. Telegram listener

Telegram listener в MVP работает через long polling.

Он должен поддерживать один или несколько ботов, статически перечисленных в конфигурации. Для каждого update listener формирует событие с основными полями фильтрации и исходным Telegram update.

Токены ботов передаются через явно указанные ссылки на переменные окружения или файлы. Динамическое добавление ботов и webhook-режим в MVP не входят.

### 5. Cron listener

Cron listener использует статический список расписаний из конфигурации демона.

Каждое расписание имеет уникальный `schedule_id`. Событие должно содержать как минимум `schedule_id`, плановое время срабатывания и фактическое время создания события.

MVP использует один документированный формат cron-выражений и явную timezone. Восстановление пропущенных срабатываний после остановки демона не выполняется.

### 6. Pipeline Executor

Каждое принятое событие передаётся в `guard` каждого активного пайплайна.

Минимальная семантика выполнения:

1. `guard` запускается первым;
2. exit code `0` означает принять событие;
3. exit code `1` означает штатно отклонить событие;
4. любой другой exit code, невозможность запуска или timeout считаются ошибкой guard;
5. после принятия события шаги выполняются строго последовательно;
6. первый неуспешный шаг останавливает только текущий запуск пайплайна;
7. отказ, ошибка или timeout одного пайплайна не влияют на остальные пайплайны.

Для одного пайплайна запуски сериализуются, чтобы его постоянное состояние не повреждалось конкурентными командами. Разные пайплайны могут выполняться параллельно в пределах глобального лимита.

Все команды запускаются через один общий process runner. Команда задаётся как `argv`; использование shell выполняется явно, например через `/bin/sh -lc`.

### 7. Минимальный manifest пайплайна

Manifest пайплайна должен содержать только:

- версию формата;
- уникальный `id`;
- `enabled`;
- команду и timeout для `guard`;
- упорядоченный список шагов;
- `id`, команду и timeout каждого шага;
- переменные окружения;
- явно разрешённые ссылки на secrets из окружения или файлов;
- связанные ресурсы пайплайна.

Retry, deduplication и настраиваемые error policies в manifest MVP не входят. Политика выполнения фиксирована правилами этого документа.

### 8. Файловое окружение запуска

Для каждого запуска создаётся отдельная временная директория. Рабочей директорией команд является эта директория.

Executor предоставляет командам как минимум:

- `event.json` с исходным событием;
- путь к ресурсам установленного пайплайна;
- постоянную директорию состояния текущего пайплайна;
- общую постоянную директорию экземпляра;
- идентификаторы события, запуска, пайплайна и шага через переменные окружения.

Ресурсы установленного пайплайна считаются read-only. Временная директория, состояние пайплайна и общая директория доступны для записи. После завершения запуска временная директория удаляется.

Специального API обмена результатами между шагами нет: шаги читают и записывают файлы в общей временной директории запуска.

### 9. Очередь и гарантии доставки

Между listeners и executor используется ограниченная очередь в памяти. Размер очереди и общий предел параллельных процессов задаются конфигурацией.

MVP не предоставляет durable queue и не обещает `at-most-once`, `at-least-once` или `exactly-once`. События могут быть потеряны при аварийном завершении, а внешний источник может доставить дубликат. Эти ограничения должны быть явно отражены в документации и логах.

## Что не входит

В MVP не входят:

- импорт и экспорт пайплайнов;
- загрузка пайплайнов из GitHub;
- registry или каталог официальных пайплайнов;
- `pipeline-building-pipeline`;
- `install.sh`, self-update, update check и rollback;
- UI, admin API и удалённое управление;
- hot reload конфигурации и динамическое подключение listeners;
- durable queue, replay и база истории запусков;
- retries, deduplication и exactly-once semantics;
- DAG, параллельные шаги и условные переходы внутри пайплайна;
- специальный API передачи результатов между шагами;
- встроенные интеграции, LLM-функции и типизированные виды шагов;
- sandbox, контейнеризация, ограничение системных ресурсов и изоляция недоверенных пайплайнов;
- собственное хранилище secrets;
- multi-node, high availability и распределённое выполнение.

Пользовательские команды выполняются с правами процесса Boarkshop. В MVP пайплайны считаются доверенными.

## Критерии готовности

MVP считается завершённым, когда автоматические end-to-end тесты подтверждают, что:

1. событие каждого из трёх listeners доходит до guard;
2. один event может быть принят несколькими пайплайнами;
3. отклонение или ошибка одного пайплайна не мешает другим;
4. следующий шаг может прочитать файл, созданный предыдущим шагом;
5. состояние пайплайна и общая директория сохраняются после перезапуска демона;
6. timeout завершает зависший процесс и останавливает только соответствующий запуск;
7. лимиты очереди и параллелизма не позволяют создавать неограниченное число процессов;
8. executor не содержит ветвлений по `source` и одинаково выполняет события всех listeners.

## Инварианты для последующего расширения

Чтобы остальная функциональность добавлялась без переделки ядра, первая реализация должна сохранить следующие границы:

- listener взаимодействует с runtime только через общий интерфейс публикации события;
- executor не зависит от схем HTTP, Telegram или Cron;
- схемы событий и manifest пайплайна версионируются;
- очередь и хранилище состояния скрыты за интерфейсами и могут быть заменены на durable реализации;
- `guard` остаётся окончательным механизмом выбора пайплайна, даже если позже появятся индексы или предварительная маршрутизация;
- все команды запускаются через единый process runner;
- управление пайплайнами, secrets, обновления и дистрибуция остаются внешними слоями относительно executor.
