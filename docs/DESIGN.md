# Boarkshop

## Назначение

Boarkshop — self-hosted демон, который принимает события из разных источников и обрабатывает их через пользовательские пайплайны.

Ядро не содержит знаний о конкретных внешних сервисах и их бизнес-логике. Оно отвечает только за получение событий и выполнение подходящих пайплайнов. Любая специфичная интеграция описывается пользователем на уровне пайплайна.

LLM или Codex не являются обязательной частью системы. С точки зрения Boarkshop это обычные исполняемые команды, которые могут использоваться в отдельных шагах.

## Технология и дистрибуция

Репозиторий проекта:

[github.com/boarkshop/boarkshop](https://github.com/boarkshop/boarkshop)

Ядро Boarkshop разрабатывается на Go и распространяется как автономный бинарный файл.

Для запуска самого Boarkshop на целевой машине не требуются:

* Go;
* Node.js;
* npm;
* отдельный runtime.

Пользовательские пайплайны при этом могут зависеть от внешних программ, интерпретаторов или CLI-инструментов, если соответствующие команды используются в их шагах. Эти зависимости не являются зависимостями ядра Boarkshop.

Пока у проекта нет собственного сайта, установочный скрипт и бинарные релизы размещаются на GitHub.

Целевая команда установки:

```bash
curl -fsSL https://raw.githubusercontent.com/boarkshop/boarkshop/main/install.sh | sh
```

Основной CLI:

```bash
boarkshop
```

Обновление до актуальной версии:

```bash
boarkshop update
```

Проверка наличия обновления без его установки:

```bash
boarkshop update --check
```

Возврат к предыдущей установленной версии:

```bash
boarkshop rollback
```

Обновление ядра не должно затрагивать:

* конфигурацию экземпляра;
* установленные пайплайны;
* ресурсы пайплайнов;
* постоянные данные пайплайнов;
* общие постоянные данные сервера.

По умолчанию обновления запускаются пользователем вручную. Автоматическое обновление может быть добавлено позднее как опциональный режим.

На текущем этапе репозиторий пуст. Команда установки станет рабочей после добавления `install.sh` и публикации первого GitHub Release.

## Демон

Демон выполняет две основные функции.

### 1. Слушает разные источники

Boarkshop поддерживает следующие типы источников:

* HTTP;
* Telegram;
* Cron.

HTTP listener представляет собой generic HTTP server. Он принимает запросы с любыми HTTP-методами и на любых путях. Сам HTTP listener не пытается определить назначение запроса или сопоставить его с конкретной интеграцией.

Telegram listener принимает обновления от одного или нескольких настроенных Telegram-ботов. Конкретный механизм динамического подключения новых ботов будет определён отдельно.

Cron создаёт события по заданному расписанию.

#### Формирование события

Каждый listener преобразует входящий запрос, обновление или срабатывание расписания в объект события и передаёт его в `guard` каждого активного пайплайна.

Объект события не ограничивается универсальной структурой вида:

```json
{
  "source": "http",
  "payload": {}
}
```

Вместо этого каждый listener определяет собственную транспортно-специфичную схему события.

Поля, характерные для всех событий конкретного listener-а и часто используемые для фильтрации, должны извлекаться заранее и размещаться непосредственно в корне объекта события. Остальные метаданные и исходные данные могут храниться во вложенных объектах.

Например, HTTP-событие может выглядеть следующим образом:

```json
{
  "source": "http",
  "event_id": "01J...",
  "received_at": "2026-07-11T12:00:00Z",
  "method": "POST",
  "path": "/webhooks/github",
  "request": {
    "query": {},
    "headers": {},
    "body": {},
    "remote_address": "192.0.2.1"
  }
}
```

Telegram-событие может иметь другую структуру:

```json
{
  "source": "telegram",
  "event_id": "01J...",
  "received_at": "2026-07-11T12:00:00Z",
  "bot_id": "main-bot",
  "update_type": "message",
  "chat_id": 123456,
  "user_id": 789012,
  "telegram": {
    "update": {}
  }
}
```

Эти примеры не фиксируют окончательные схемы.

При реализации каждого listener-а необходимо отдельно определить:

* какие поля присутствуют во всех его событиях;
* какие поля наиболее полезны для фильтрации;
* какие поля должны находиться в корне события;
* какие поля являются опциональными;
* где хранится исходное транспортное представление события.

Схема должна быть стабильной и предсказуемой для всех событий одного listener-а. Обязательность и допустимая пустота каждого поля являются частью контракта listener-а.

Listener выполняет транспортную нормализацию, но не бизнес-интерпретацию. Например, HTTP listener извлекает `method` и `path`, а Telegram listener может извлекать `bot_id`, `update_type` и `chat_id`. При этом listener не определяет, является ли HTTP-запрос событием GitHub, не проверяет подпись конкретного сервиса и не решает, какой пайплайн должен его обработать.

Цель такой структуры — сделать `guard` максимально простым. Типичный `guard` должен иметь возможность выглядеть примерно так:

```text
if source == "telegram" && chat_id in authorized_chats => accept
```

или:

```text
if source == "http" && method == "POST" && path == "/webhooks/github" => accept
```

Одно событие передаётся всем активным пайплайнам и может быть принято несколькими из них.

### 2. Выполняет пользовательские пайплайны

Для каждого активного пайплайна Boarkshop:

1. запускает его первый шаг — `guard`;
2. получает решение, должен ли пайплайн обработать событие;
3. при положительном решении последовательно выполняет остальные шаги;
4. при отрицательном решении завершает выполнение этого пайплайна.

Отказ одного пайплайна не влияет на проверку и выполнение остальных пайплайнов.

## Pipeline

Пайплайн представляет собой упорядоченную последовательность команд:

```text
guard → step 1 → step 2 → ... → step N
```

`guard` — обязательный первый шаг пайплайна. Он получает сформированный listener-ом объект события и определяет, должен ли данный пайплайн продолжить выполнение.

Логика `guard` может быть произвольно сложной, но структура событий должна позволять реализовывать большинство `guard`-ов через простые проверки полей верхнего уровня.

`guard` реализуется такой же командой или программой, как и любой другой шаг пайплайна.

Если `guard` отклоняет событие, остальные шаги этого пайплайна не запускаются.

## Step

Каждый шаг сводится к одному и тому же базовому действию: запуску указанной команды в подготовленном окружении.

Командой может быть любой исполняемый процесс:

* shell-команда;
* Python-скрипт;
* бинарный файл;
* вызов Codex или другой LLM;
* пользовательский CLI-инструмент;
* любой другой доступный исполняемый процесс.

С шагом также могут быть связаны необходимые ему ресурсы:

* код;
* промпты;
* конфигурации;
* документы;
* бинарные файлы;
* шаблоны;
* другие файлы.

Boarkshop не вводит отдельные типы шагов для скриптов, LLM или внешних инструментов. С точки зрения runtime все они являются запуском команды.

Шаги не передают друг другу результаты через специальный внутренний API. Если шаг зависит от результатов предыдущих шагов, он должен читать их из доступных директорий. Аналогично, результат для следующих шагов должен быть записан в файл.

## Директории

Во время выполнения команд Boarkshop предоставляет три уровня файлового состояния.

### Временная директория запуска

Для каждого запуска пайплайна создаётся отдельная временная директория.

В ней могут находиться:

* исходный объект события;
* ресурсы, необходимые шагам;
* промежуточные результаты;
* файлы, созданные командами;
* итоговые артефакты текущего запуска.

Все шаги одного запуска используют одну временную директорию и могут передавать результаты друг другу через файлы в ней.

После завершения запуска временная директория удаляется.

### Постоянная директория пайплайна

Каждый пайплайн имеет собственную постоянную директорию для хранения состояния между запусками.

Команды пайплайна могут читать из неё и записывать в неё. Содержимое этой директории не удаляется после завершения запуска.

Она может использоваться для:

* накопленного состояния;
* локальной базы данных;
* кэша;
* истории, необходимой самому пайплайну;
* файлов, которые должны быть доступны следующим запускам.

Эта директория принадлежит конкретному пайплайну и не используется другими пайплайнами по умолчанию.

### Общая директория сервера

Экземпляр Boarkshop также имеет постоянную общую директорию.

Она доступна разным пайплайнам и может использоваться для файлов и состояния, общих для всего экземпляра Boarkshop.

Содержимое общей директории сохраняется между запусками пайплайнов, перезапусками демона и обновлениями ядра.

## Конфигурация пайплайна

Пайплайн хранится как директория, содержащая машиночитаемую конфигурацию и связанные с ней ресурсы.

Описание пайплайна включает:

* уникальный идентификатор;
* состояние активности;
* команду `guard`;
* упорядоченную последовательность шагов;
* команды и ресурсы шагов;
* переменные окружения;
* доступные secrets;
* тайм-ауты;
* правила обработки ошибок;
* правила повторных запусков;
* правила дедупликации событий.

`guard` является первым элементом последовательности выполнения, а не отдельным типом runtime-компонента.

## Источники пайплайнов

Базовая установка Boarkshop не содержит установленных или активных пайплайнов. Runtime запускается пустым, а необходимые пайплайны устанавливаются пользователем.

Пайплайн может быть:

* создан локально;
* сгенерирован другим пайплайном или инструментом;
* импортирован из GitHub;
* экспортирован для хранения или публикации.

### Импорт

Пайплайн можно импортировать из GitHub-репозитория:

```bash
boarkshop import git@github.com:boarkshop/pipelines --path useful-pipeline/v1/
```

Параметр `--path` является опциональным. Если он указан, Boarkshop:

1. клонирует репозиторий;
2. находит указанный путь внутри репозитория;
3. копирует содержимое этого пути в локальное хранилище пайплайнов;
4. устанавливает пайплайн в текущий экземпляр Boarkshop.

Если `--path` не указан, источником пайплайна считается корень репозитория.

### Экспорт

Установленный пайплайн можно экспортировать в локальную директорию:

```bash
boarkshop export -n some-pipeline -p ~/mypipelines/some-pipeline/
```

Экспорт включает конфигурацию пайплайна и необходимые ему ресурсы. Постоянные runtime-данные пайплайна по умолчанию не входят в экспорт.

### Официальные пайплайны

Проект Boarkshop может предоставлять набор официальных дополнительных пайплайнов. Они не являются частью ядра и не устанавливаются автоматически, но могут быть импортированы обычным способом.

Одним из основных официальных пайплайнов будет `pipeline-building-pipeline`.

Он работает через Telegram-бота и позволяет по пользовательскому описанию:

* создавать новые пайплайны;
* изменять существующие пайплайны;
* генерировать команды, скрипты, промпты и другие ресурсы;
* показывать предполагаемые изменения;
* применять изменения после явного подтверждения пользователя.

С точки зрения ядра `pipeline-building-pipeline` является обычным пользовательским пайплайном.

## Граница системы

```text
Boarkshop
├── listeners
│   ├── generic HTTP
│   ├── Telegram
│   └── Cron
└── pipeline executor
    └── guard → step 1 → step 2 → ... → step N
```

Ядро слушает события, выполняет их транспортную нормализацию и запускает команды. Вся интеграционная и бизнес-логика находится внутри устанавливаемых пользователем пайплайнов.
