# Boarkshop

## Назначение

Boarkshop — self-hosted демон, который принимает события из разных источников и обрабатывает их через пользовательские пайплайны.

Ядро не содержит знаний о конкретных внешних сервисах и их бизнес-логике. Оно отвечает за:

* приём событий;
* формирование удобного объекта события;
* передачу события активным пайплайнам;
* последовательное выполнение команд пайплайна.

Любая специфичная интеграция описывается пользователем на уровне пайплайна.

LLM или Codex не являются обязательной частью системы. С точки зрения Boarkshop это обычные исполняемые команды, которые могут использоваться в отдельных шагах.

## Технология и дистрибуция

Репозиторий проекта:

[github.com/boarkshop/boarkshop](https://github.com/boarkshop/boarkshop)

Ядро Boarkshop разрабатывается на Go и распространяется как автономный бинарный файл.

Для запуска самого Boarkshop на целевой машине не требуются:

* Go;
* Node.js;
* npm;
* отдельный runtime.

Пользовательские пайплайны при этом могут зависеть от внешних программ, интерпретаторов или CLI-инструментов, если соответствующие команды используются в их шагах. Такие зависимости являются зависимостями конкретного пайплайна, а не ядра Boarkshop.

Пока у проекта нет собственного сайта, установочный скрипт и бинарные релизы размещаются на GitHub.

Целевая команда установки:

```bash
curl -fsSL https://raw.githubusercontent.com/boarkshop/boarkshop/main/install.sh | sh
```

Основной CLI:

```bash
boarkshop
```

Обновление до актуальной версии:

```bash
boarkshop update
```

Проверка наличия обновления без его установки:

```bash
boarkshop update --check
```

Возврат к предыдущей установленной версии:

```bash
boarkshop rollback
```

Обновление ядра не должно затрагивать:

* конфигурацию экземпляра;
* установленные пайплайны;
* ресурсы пайплайнов;
* постоянные данные пайплайнов;
* общие постоянные данные сервера.

По умолчанию обновления запускаются пользователем вручную. Автоматическое обновление может быть добавлено позднее как опциональный режим.

На текущем этапе репозиторий пуст. Команда установки станет рабочей после добавления `install.sh` и публикации первого GitHub Release.

## Демон

Демон выполняет две основные функции.

### 1. Слушает разные источники

Boarkshop поддерживает следующие типы источников:

* HTTP;
* Telegram;
* Cron.

HTTP listener представляет собой generic HTTP server. Он принимает запросы с любыми HTTP-методами и на любых путях.

Сам HTTP listener не пытается определить назначение запроса, тип внешнего сервиса или пайплайн, который должен его обработать.

Telegram listener принимает обновления от одного или нескольких настроенных Telegram-ботов. Один экземпляр Boarkshop должен поддерживать несколько ботов. Механизм динамического подключения новых ботов без перезапуска будет определён позднее.

Cron создаёт события по заданному расписанию.

#### Формирование события

Каждый listener преобразует входящий запрос, Telegram update или срабатывание расписания в объект события.

Этот объект передаётся в первый шаг каждого активного пайплайна.

Объект события не ограничивается универсальной структурой вида:

```json
{
  "source": "http",
  "payload": {}
}
```

Каждый listener определяет собственную транспортно-специфичную схему события.

Поля, которые характерны для событий данного listener-а и часто используются при фильтрации, должны извлекаться заранее и размещаться непосредственно в корне объекта. Остальные данные могут храниться во вложенных объектах.

Цель такого подхода — минимизировать код в `guard`.

Например, HTTP-событие может выглядеть следующим образом:

```json
{
  "source": "http",
  "event_id": "01J...",
  "received_at": "2026-07-11T12:00:00Z",
  "method": "POST",
  "path": "/webhooks/github",
  "request": {
    "query": {},
    "headers": {},
    "body": {},
    "remote_address": "192.0.2.1"
  }
}
```

Telegram-событие может выглядеть следующим образом:

```json
{
  "source": "telegram",
  "event_id": "01J...",
  "received_at": "2026-07-11T12:00:00Z",
  "bot_id": "main-bot",
  "update_type": "message",
  "chat_id": 123456,
  "user_id": 789012,
  "telegram": {
    "update": {}
  }
}
```

Эти примеры не фиксируют окончательные схемы.

При реализации каждого listener-а необходимо отдельно определить:

* какие поля присутствуют в его событиях;
* какие поля гарантированно присутствуют во всех событиях;
* какие поля являются опциональными;
* какие поля наиболее полезны для фильтрации;
* какие поля должны находиться в корне объекта;
* где хранится исходное транспортное представление события.

Схема каждого listener-а должна быть стабильной и предсказуемой. Обязательность, тип и допустимая пустота каждого поля являются частью контракта listener-а.

Listener выполняет транспортную нормализацию, но не бизнес-интерпретацию.

Например, HTTP listener извлекает `method` и `path`, а Telegram listener может извлекать `bot_id`, `update_type`, `chat_id` и `user_id`.

При этом listener не должен:

* определять, является ли HTTP-запрос событием GitHub, Stripe или другого сервиса;
* проверять подпись конкретного внешнего сервиса;
* интерпретировать бизнес-смысл payload;
* решать, какой пайплайн должен обработать событие.

Типичный `guard` должен иметь возможность выглядеть примерно так:

```text
if source == "telegram" && chat_id in authorized_chats => accept
```

или:

```text
if source == "http" && method == "POST" && path == "/webhooks/github" => accept
```

Одно событие передаётся всем активным пайплайнам и может быть принято несколькими из них.

### 2. Выполняет пользовательские пайплайны

Для каждого активного пайплайна Boarkshop:

1. запускает его первый шаг — `guard`;
2. получает решение, должен ли пайплайн обработать событие;
3. при положительном решении последовательно выполняет остальные шаги;
4. при отрицательном решении завершает выполнение этого пайплайна.

Решение одного пайплайна не влияет на проверку и выполнение остальных пайплайнов.

## Pipeline

Пайплайн представляет собой упорядоченную последовательность команд:

```text
guard → step 1 → step 2 → ... → step N
```

`guard` — обязательный первый шаг пайплайна. Он получает сформированный listener-ом объект события и определяет, должен ли данный пайплайн продолжить выполнение.

Логика `guard` может быть произвольно сложной, но структура событий должна позволять реализовывать большинство `guard`-ов через простые проверки полей верхнего уровня.

`guard` реализуется такой же командой или программой, как и любой другой шаг пайплайна.

Если `guard` отклоняет событие, остальные шаги этого пайплайна не запускаются.

Пайплайн является не отдельным YAML-файлом, а директорией с конфигурацией и связанными с ней файлами.

YAML-файл описывает последовательность выполнения, но сам пайплайн также может содержать:

* скрипты;
* промпты;
* шаблоны;
* документы;
* локальные конфигурации;
* бинарные файлы;
* токены и другие учётные данные;
* любые другие вспомогательные ресурсы.

## Step

Каждый шаг сводится к одному базовому действию: запуску указанной команды в подготовленном окружении.

Командой может быть любой исполняемый процесс:

* shell-команда;
* Python-скрипт;
* бинарный файл;
* вызов Codex или другой LLM;
* пользовательский CLI-инструмент;
* любой другой доступный исполняемый процесс.

С шагом могут быть связаны необходимые ему ресурсы:

* код;
* промпты;
* конфигурации;
* документы;
* бинарные файлы;
* шаблоны;
* токены и другие учётные данные;
* любые другие вспомогательные файлы.

Boarkshop не вводит отдельные типы шагов для скриптов, LLM или внешних инструментов. С точки зрения runtime все они являются запуском команды.

### Доступ к директориям

Перед запуском каждой команды Boarkshop должен предоставлять ей пути как минимум к следующим директориям:

* корневой директории установленного пайплайна;
* временной директории текущего запуска;
* постоянной директории данных текущего пайплайна;
* общей постоянной директории экземпляра Boarkshop.

Благодаря этому `guard` и остальные шаги могут обращаться к скриптам, промптам, конфигурациям и другим ресурсам внутри директории пайплайна.

Например, команда должна иметь возможность запустить файл:

```text
<pipeline_dir>/scripts/process.py
```

Конкретный способ передачи путей пока не фиксируется. Они могут быть доступны через:

* подстановку в описании команды;
* переменные окружения;
* другой стабильный механизм runtime.

Возможный, но не утверждённый формат:

```yaml
cmd: python {{ pipeline_dir }}/scripts/process.py
```

Синтаксис `{{ pipeline_dir }}` приведён только как пример. Требование заключается в том, что каждая команда должна иметь надёжный доступ к абсолютному пути директории своего пайплайна.

Аналогичная возможность должна существовать для временной, постоянной и общей директорий.

### Обмен результатами

Шаги не передают друг другу результаты через специальный внутренний API.

Если шаг зависит от результатов предыдущих шагов, он должен читать их из временной директории текущего запуска.

Результаты, предназначенные для следующих шагов, также должны записываться в файлы внутри этой директории.

Таким образом:

* директория установленного пайплайна используется для доступа к конфигурации и ресурсам;
* временная директория используется для обмена данными внутри одного запуска;
* постоянная директория пайплайна используется для состояния между запусками;
* общая директория используется для состояния, доступного нескольким пайплайнам.

Система не навязывает формат файлов или внутренние соглашения между шагами. Их определяет автор пайплайна.

## Локальная файловая модель

Конфигурации, ресурсы и данные Boarkshop хранятся непосредственно на сервере, где запущен демон.

Ядро Boarkshop:

* не отправляет содержимое этих директорий во внешнее хранилище;
* не синхронизирует их с облачным сервисом;
* не требует внешнего control plane;
* не требует внешней базы данных для основной работы.

Файлы могут покинуть сервер только в результате явного действия пользователя или команды внутри самого пайплайна.

Поскольку шагом может быть любая исполняемая команда, сам пайплайн технически может отправлять файлы или данные по сети. Boarkshop не выполняет такую отправку самостоятельно.

Boarkshop использует четыре типа директорий.

### Директория установленного пайплайна

Каждый установленный пайплайн хранится в отдельной локальной директории.

Эта директория является полным описанием пайплайна и может содержать:

* основной YAML-файл конфигурации;
* дополнительные конфигурации;
* guard-скрипты;
* скрипты шагов;
* промпты;
* шаблоны;
* документы;
* бинарные файлы;
* токены и другие учётные данные;
* любые другие необходимые ресурсы.

Пример структуры:

```text
some-pipeline/
├── pipeline.yaml
├── scripts/
│   ├── guard.sh
│   └── process.py
├── prompts/
│   └── system.md
├── config/
│   └── settings.json
└── resources/
    └── telegram-token
```

Это пример, а не обязательная структура.

Конкретные имена директорий и файлов определяются автором пайплайна. Конфигурация пайплайна может ссылаться на файлы относительно корня этой директории.

Директория установленного пайплайна сохраняется между запусками и обновлениями ядра Boarkshop.

### Временная директория запуска

Для каждого запуска пайплайна создаётся отдельная временная директория.

Запуск включает выполнение `guard` и, если событие принято, последующих шагов.

Во временной директории могут находиться:

* исходный объект события;
* промежуточные результаты;
* файлы, созданные командами;
* итоговые артефакты текущего запуска.

Все шаги одного запуска используют одну временную директорию и могут передавать результаты друг другу через файлы в ней.

После завершения запуска временная директория удаляется.

### Постоянная директория пайплайна

Каждый пайплайн имеет отдельную постоянную директорию для изменяемого состояния между запусками.

Команды пайплайна могут читать из неё и записывать в неё. Содержимое этой директории не удаляется после завершения запуска.

Она может использоваться для:

* накопленного состояния;
* локальной базы данных;
* кэша;
* истории, необходимой самому пайплайну;
* файлов, которые должны быть доступны следующим запускам.

Постоянная директория данных отделена от директории с конфигурацией и ресурсами пайплайна. Это позволяет обновлять или переустанавливать описание пайплайна, не удаляя накопленное им состояние.

По умолчанию эта директория принадлежит только конкретному пайплайну.

### Общая директория сервера

Экземпляр Boarkshop имеет постоянную общую директорию.

Она доступна разным пайплайнам и может использоваться для файлов и состояния, общих для всего экземпляра Boarkshop.

Содержимое общей директории сохраняется между:

* запусками пайплайнов;
* перезапусками демона;
* обновлениями ядра.

## Секреты

Ресурсы пайплайна могут содержать секретные данные.

Например, токен Telegram-бота может храниться в отдельном файле внутри локальной директории пайплайна:

```text
some-pipeline/resources/telegram-token
```

В первой версии Boarkshop такие данные хранятся в plaintext. Встроенного шифрования секретов в MVP нет.

Защита этих данных на данном этапе обеспечивается:

* файловыми правами операционной системы;
* ограничением доступа к серверу;
* ограничением доступа к директориям Boarkshop;
* аккуратным обращением с импортом и экспортом пайплайнов.

В будущих версиях должна быть добавлена возможность шифрования секретных ресурсов на диске.

Механизм шифрования не входит в MVP и будет спроектирован отдельно.

## Конфигурация пайплайна

Пайплайн хранится как директория, содержащая машиночитаемую конфигурацию и связанные с ней ресурсы.

Основной YAML-файл является точкой входа в описание пайплайна, но не обязан содержать весь его код и данные.

Команды могут ссылаться на любые вспомогательные файлы внутри директории пайплайна.

Описание пайплайна включает:

* уникальный идентификатор;
* состояние активности;
* команду `guard`;
* упорядоченную последовательность шагов;
* команды шагов;
* пути к ресурсам;
* переменные окружения;
* тайм-ауты;
* правила обработки ошибок;
* правила повторных запусков;
* правила дедупликации событий.

`guard` является первым элементом последовательности выполнения, а не отдельным типом runtime-компонента.

## Источники пайплайнов

Базовая установка Boarkshop не содержит установленных или активных пайплайнов.

Runtime запускается пустым, а необходимые пайплайны устанавливаются пользователем.

Пайплайн может быть:

* создан локально;
* сгенерирован другим пайплайном или инструментом;
* импортирован из GitHub;
* экспортирован для хранения или публикации.

### Импорт

Пайплайн можно импортировать из GitHub-репозитория:

```bash
boarkshop import git@github.com:boarkshop/pipelines --path useful-pipeline/v1/
```

Параметр `--path` является опциональным.

Если он указан, Boarkshop:

1. клонирует репозиторий;
2. находит указанный путь внутри репозитория;
3. копирует всю директорию пайплайна, включая конфигурацию и вспомогательные файлы, в локальное хранилище;
4. устанавливает пайплайн в текущий экземпляр Boarkshop.

Если `--path` не указан, источником пайплайна считается корень репозитория.

После импорта пайплайн полностью хранится локально. Для его дальнейшего выполнения доступ к исходному GitHub-репозиторию не требуется.

### Экспорт

Установленный пайплайн можно экспортировать в локальную директорию:

```bash
boarkshop export -n some-pipeline -p ~/mypipelines/some-pipeline/
```

Экспорт копирует описание пайплайна как директорию, включая:

* YAML-конфигурацию;
* скрипты;
* промпты;
* конфигурационные файлы;
* бинарные файлы;
* другие связанные ресурсы.

По умолчанию в экспорт не входят:

* временные директории запусков;
* постоянные данные пайплайна;
* общие данные сервера.

Поскольку в MVP ресурсы могут содержать plaintext-токены и другие секреты, экспортированная директория должна считаться чувствительной.

Если секретный файл находится внутри экспортируемой директории пайплайна, он также попадёт в экспорт.

### Официальные пайплайны

Проект Boarkshop может предоставлять набор официальных дополнительных пайплайнов.

Они не являются частью ядра и не устанавливаются автоматически, но могут быть импортированы обычным способом.

Одним из основных официальных пайплайнов будет `pipeline-building-pipeline`.

Он работает через Telegram-бота и позволяет по пользовательскому описанию:

* создавать новые пайплайны;
* изменять существующие пайплайны;
* генерировать команды, скрипты, промпты и другие ресурсы;
* показывать предполагаемые изменения;
* применять изменения после явного подтверждения пользователя.

С точки зрения ядра `pipeline-building-pipeline` является обычным пользовательским пайплайном.

## Граница системы

```text
Boarkshop
├── listeners
│   ├── generic HTTP
│   ├── Telegram
│   └── Cron
└── pipeline executor
    └── guard → step 1 → step 2 → ... → step N
```

Ядро Boarkshop:

* слушает события;
* преобразует их в удобные транспортно-специфичные объекты;
* передаёт события активным пайплайнам;
* последовательно запускает команды.

Конфигурации, ресурсы и данные хранятся локально на сервере Boarkshop.

Вся интеграционная и бизнес-логика находится внутри устанавливаемых пользователем пайплайнов.
